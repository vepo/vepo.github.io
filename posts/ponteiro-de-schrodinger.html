Foi um tormento que eu enfrentei com um aperitivo num sábado de manhã. O quero falar aqui do Ponteiro de Schrödinger. Mas você deve estar se perguntando o que é o tal Ponteiro de Schrödinger uma vez que nunca deve ter ouvido esse termo porque eu acabei de inventar. Isso porque por mais que as pessoas falem que "ponteiro não existe na minha linguagem", (i) ele pode realmente não existir, (ii) o conceito dele pode existir mas a linguagem gerencia isso de forma eficiente E (iii) as pessoas podem recriar ponteiro e fazer códigos tão esdruxulos que fariam Ada Lovelace desistir desse negócio de programação e se dedicar a poesia e a bohemia como o pai dela.

Então vamos comecar por cada um desses pontos

(i) Ponteiros realmente não existem na sua linguagem

Eu trabalho com Java e a última vez que vi um ponteiro foi antes de 2015 enquanto trabalhava com C e C++. Em Java e em muitas das linguagens modernas, o conceito de ponteiro faz parte do dia a dia de um desenvolvedor. Quer dizer…​ Na verdade faz, mas a linguagem faz de tudo pra esquecer isso e é preciso.

Java é uma linguagem que surgiu em uma época que Orientação a Objetos era a panaceia para todos os problemas na computação. Se falava de Objetos Distribuídos como se eles fossem a solução de problemas arquiteturais e o último Design Pattern a existir (ok, Design Patterns ainda estavam engatinhando). Por isso Orientação a Objetos tem um lugar central na sua linguagem e por isso que existe uma exception chamada NullPointerException.

Apesar de existir a exceção, as linguagens fizeram ao máximo para esconder a realidade dos ponteiros. As vezes usam outro nome como TypeError, as vezes criam o conceito de classe e instância, criam a abstração chamada Optional, mas elas fazem de tudo para esconder a triste realidade que tudo é um endereço de memória cheia de 1 e 0.

Pointeiros existem em Javascript e eu posso provar!

Toda a discussão sobre ponteiros se dá porque precisamos discutir se (i) existe um valor e (ii) se, caso esse valor exista, dois valores são iguais ou não. Java abstraiu isso criando métodos dentro da classe (um conceito não criado pelo Java) que as vezes precisamos nos preocupar.

Por exemplo, quando precisamos implementar os métodos hashCode e equals para que nossa classe possa ser eficientemente gerenciada em um Map ou um Set, isso não é um preciosismo ou uma frescura. É uma necessidade. Em alguns casos, valores que são lidos em locais diferentes são iguais e precisamos identificar isso.

Dessa forma quem criou as linguagens com certeza se deparou com o dilema entre igualdade e identidade. Dois objetos idênticos podem ser iguais? E dois iguais podem não ser idênticos?

Essa questão é muito facilmente resolvida se eu perguntar: os funcionários A e B ganham um salário X, se eu aumentar o salário de B eu aumento o salário de A? Observa que essa igualdade não é identidade. Os salários são iguais, mas não são os mesmos. Logo esse valor não pode ser armazenado na mesma linha de um banco de dados ou em um mesmo objeto. Eles são iguais, mas não são idênticos.

Mas há a grande falha no design das linguagens que é a questão da ausência. Programador é muito bom com a lógica booleana ao pensar na conceito de verdadeiro ou falso, mas a lógica computacional possui também o conceito de ausente (que é diferente de inexistente, ou undefined). Por exemplo, um valor booleano pode ser true, false ou null. E como representamos o nulo sem dizer que o nulo existe? Por isso ainda usamos o null que vem do C, vem do baixo nível e é tão difícil de ser compreendido.

(ii) O Conceito de Ponteiro existe, mas ele é gerenciado pela linguagem

O conceito de null vem da história da computação, mais especificamente do ALGOL, criado 1965. Eu não diria que foi um erro, mas uma má ideia e há explicação para isso.

Para entender o conceito de null é preciso entender como a memória funciona e abstrações surgiram na programação. Eu entendo bem disso porque fiz Engenharia de Computação e grande parte do curso eu programei Assembly, não recomendo.

Sendo bem genérico, um computador é uma unidade de processamento ligada a um barramento que acessa dispositivos. A CPU possui uma memória interna (cache e registradores), mas ela sempre acessa a memória e outros dispositivos. Se você deseja criar um dispositivo IoT que lerá um sensor, ele se comportará como uma posição de memória e você acessará ele, mas para isso precisa construir um driver para expor isso em "alto nível". Essa é uma discussão muito interessante, mas limitante, fica complicado demais falar de design de código pensando no baixo nível.

Vamos pensar em termos de Strings. A palavra String significa cadeia, ou seja, uma String é uma Cadeia de Caracteres. Dessa forma, vamos pensar na frase "A CPU possui uma memória interna" retirada do paragrafo anterior. Se você inspecionar a memória do seu computador ao ler esse post, verá algo como a imagem abaixo.

Esse post, durante a escrita, lido com o programa xxd.

Observa que as duas referências ao trecho existem como duplicadas na memória e que ambas possuem endereço de memória diferentes (0000186a e 00001600). Logo se eu pegar um objeto String com esse valor e chamar o método toUpperCase(), o que vai acontecer?

Esse post, durante a escrita, lido com o programa xxd sem agrupar bytes.

Uma nova String com o valor "A CPU POSSUI UMA MEMÓRIA INTERNA" será criada! Isso porque o Java não quer que o programador se preocupe com o baixo nível, o que muitas vezes pode tornar o programa menos eficiente. Em C, eu poderia implementar a função toUpperCase que seria muito mais eficiente e usaria menos memória, mas alteraria somente o valor desejado.

toUpperCase "in place"
void toUpperCase(char* content, int offset, int length) {
    int index = 0;
    while (index < length) {
        if (content[offset + index] >= 'a' && content[offset + index] <= 'z') {
            content[offset + index] =  content[offset + index] - 32;
        }
        ++index;
    }
}

Observe como ela pode ser usada no trecho abaixo.

char* content = "o que acessa dispositivos. A CPU possui uma memória interna (cache e registradores), mas ela sempre ac";
printf("%s\n", content); // Imprime: o que acessa dispositivos. A CPU possui uma memória interna (cache e registradores), mas ela sempre ac
toUpperCase(content, 27, 32);
printf("%s\n", content); // Imprime: o que acessa dispositivos. A CPU POSSUI UMA MEMÓRIA INTERNA (cache e registradores), mas ela sempre ac

Talvez você ache que eu esteja viajando, mas quero mostrar que há inúmeras discussões que uma escolha arquitetural pode eliminar. Em Java, uma String é uma cadeia de caracteres, mas ela é imutável. Dessa forma, o código abaixo é seguro, pois o desenvolvedor nem se preocupa sobre qual referência de memória o código dele está sendo executado.

var user = entityManager.find(User.class, userId); // [1] {'id': 1, 'username': 'jao', 'nome': 'João', sobrenome': 'Silva'}
map.put(user.getUsername().toUpperCase(), user);   // [2] 'JAO' → {'id': 1, 'username': 'jao', 'nome': 'João', sobrenome': 'Silva'}
System.out.println(user.getUsername())             // [3] 'jao'

Foi decidido enquanto a linguagem estava sendo definida, que Strings são valores imutáveis, e isso significa que valores alterados em uma String implica na criação de outra String. É seguro, mas em alguns casos ineficiente, por isso foi necessário criar objetos como o StringBuilder e o StringBuffer.

Se você não é do mundo Java, provavelmente existe a mesma discussão na sua linguagem.

(iii) Como as pessoas reinventam os Ponteiros e o fazem de maneira errada!

Não é raro encontrarmos em Java pessoas reinventando ponteiros para suplantar "limitações" da linguagem. Quero já dizer aqui que isso não é uma limitação da linguagem, mas a linguagem preocupada que você não faça 💩, mas as pessoas continuam fazendo 💩 porque elas não sabem resolver o problema de um outro modo.

A primeira limitação que me vem a mente é a obrigatoriedade que uma variável seja final ao se usar em contextos que possivelmente podem ser acessados por mais de uma thread. Essa limitação é imposta para que o desenvolvedor não precise se preocupar com a concorrência entre as possíveis threads.

Para exemplificar, vou mostrar um exemplo bem tosco que já encontrei em diversas bases de código de diversas empresas. Vamos imaginar que temos uma lista de valores e queremos encontrar um valor especifico. Aí o desenvolvedor resolver usar lambdas, mas sem se dar o trabalho de ler muito a documentação da biblioteca padrão. Como ele implementa?

List<User> users = getAllUsers();  // isso é só um exemplo
User[] primeiroUsuarioAtivo = new User[1]; // Quem nunca fez isso?
users.forEach(user -> { // Podia ser um loop? Podia! Mas foi o exemplo que eu pensei
    if (primeiroUsuarioAtivo[0] == null && user.isAtivo()) { // Isso pode dar 💩
        primeiroUsuarioAtivo[0] = user; // Isso também pode dar 💩
    }
});
System.out.println("Primeiro usuário ativo: " + Arrays.toString(primeiroUsuarioAtivo));

O código acima compila e é sintaticamente correto, mas há diversos problemas nele! Também já quero alertar que a possibilidade de algum erro acontecer é baixa, visto que provavelmente a lista retornada por getAllUsers() não reimplementa o método forEach. Mas ninguém garante a função lambda não será executada em sequência e serialmente.

A exigência que a variáveis users e primeiroUsuarioAtivo sejam final, e na verdade é mesmo que você não defina ela como final é feita para garantir que os valores não possa ser alterados por processos concorrentes. Esse código ficaria mais "correto" se fosse sincronizado, como abaixo, mas fica ruim. 🤨

List<User> users = getAllUsers();  // isso é só um exemplo
User[] primeiroUsuarioAtivo = new User[1]; // Quem nunca fez isso?
users.forEach(user -> { // Podia ser um loop? Podia! Mas foi o exemplo que eu pensei
    synchronized (primeiroUsuarioAtivo) {
        if (primeiroUsuarioAtivo[0] == null && user.isAtivo()) { // Isso pode dar 💩
            primeiroUsuarioAtivo[0] = user; // Isso também pode dar 💩
        }
    }
});
System.out.println("Primeiro usuário ativo: " + Arrays.toString(primeiroUsuarioAtivo));

Isso não é nenhum problema com a linguagem Java. Essas limitações foram feitas para que você fizesse um bom design de código e escrevesse um código que comunique ao desenvolvedor a sua intenção. Então porque não reescrever ele como abaixo?

List<User> users = getAllUsers();
Optional<User> maybeActive = users.stream()
                                  .filter(User::isAtivo)
                                  .findFirst();
if (maybeActive.isPresent()) {
    System.out.println("Primeiro usuário ativo: " + maybeActive.get());
} else {
    System.out.println("Nenhum usuário ativo encontrado!");
}

Observe que esse código é muito mais expressivo que o anterior e também é mais eficaz! Nele não há problemas de concorrência, uma vez que isso é delegado a JVM. E não foi preciso criar um vetor para ser usado como um ponteiro informal.

Mas há outras formas que o mesmo problema pode levar a se recriar ponteiros. Faz algum tempo eu implementei uma classe que se chamava SharedMaps. A função dela era garantir que mapas fossem usados e atualizados de forma assíncrona e eficiente. Essa classe usava ReadWriteLock para garantir que leituras pudessem ser feitas em paralelo e escritas fossem feitas sincronicamente. MAS…​ como a implementação foi feita há muito tempo, o uso era pontual e ela tinha apenas dois métodos. Essa classe resolveu um grande problema nosso de não ver mais ConcurrentModificationException.

public class SharedMaps {
    private final ReadWriteLock readWriteLock = new ReentrantReadWriteLock();
    private final Lock writeLock = readWriteLock.writeLock();
    private final Lock readLock = readWriteLock.readLock();
    private final Map<Map<String, ?>> internalMaps;

    public SharedMaps() {
        this.internalMaps = new HashMap<>();
    }

    public <T> void forRead(String mapKey, Consumer<Map<String, T>> fn) {
        readLock.lock();
        try {
            fn.accept(internalMaps.computeIfAbsent(mapKey, __ -> new HashMap<>()));
        }
        finally {
            // To unlock the acquired read thread
            readLock.unlock();
        }
    }

    public <T> void forWrite(String mapKey, Consumer<Map<String, T>> fn) {
        writeLock.lock();
        try {
            fn.accept(internalMaps.computeIfAbsent(mapKey, __ -> new HashMap<>()));
        }
        finally {
            // To unlock the acquired write thread
            writeLock.unlock();
        }
    }
}

Com essa classe o desenvolvedor poderia usar o mapa de forma sincrona. A necessidade dessa classe era justificada e o uso também era pontual. Mas surgiram novas funcionalidades e em uma delas surgiu a necessidade de se retornar apenas um valor para leitura. Pelo meu entendimento, o desenvolvedor desejaria algum método como retrieveValueOrCreate(mapKey, valueKey, this::createValue) e eu implementaria ele como fiz abaixo.

public <T> T retrieveValueOrCreate(String mapKey, String valueKey, Supplier<T> supplier) {
    writeLock.lock();
    try {
        return internalMaps.computeIfAbsent(mapKey, __ -> new HashMap<>()).computeIfAbsent(valueKey, supplier);
    }
    finally {
        // To unlock the acquired write thread
        writeLock.unlock();
    }
}

MAS…​ o desenvolvedor que pegou essa atividade não pensou duas vezes. Não alterou a classe SharedMaps e nem conhecia mais interfaces da biblioteca padrão do Java além das básicas ensinadas em algum curso, e ignorou completamente as classes Supplier, Function, BiFunction ou mesmo o conceito de interface funcional. Ele simplesmente usou AtomicReference como um ponteiro.

AtomicReference<User> foundUser = new AtomicReference();
sharedMaps.forWrite(mapKey, map -> {
    User user = map.get(valueKey);
    if (user == null) {
        user = createValue();
        map.put(valueKey, user);
    }
    foundUser.set(user);
})
Objects.assertNonNull(foundUser.get());
// usava o foundUser

No fundo, o código acima é complicado e ineficiente, uma vez que a variável foundUser é usada como um ponteiro. Ele poderia ser facilmente substituido pelo código abaixo.

User foundUser = sharedMaps.retrieveValueOrCreate(mapKey, valueKey, this::createValue);
// usa o foundUser

Eu ainda preciso ressaltar que o código que eu encontrei não era tão simples e que o lambda usado não era tão eficaz. Havia muita lógica dentro do lambda o que fazia com que o código ficasse sincronizado e se tornava menos eficaz. Uma interface funcional tem que ser eficaz e não pode alterar nada externo ao seu contexto, POR ISSO QUE ELA DEVE SOMENTE ACESSAR VARIÁVEIS MARCADAS COMO final.

Conclusão

As vezes é importante saber que ponteiros existem porque eles refletem um detalhe de implementação das nossas máquinas. Esse conceito é importante para garantir eficiência e consistência dos programas, mas ele no fundo existem apenas no baixo nível. As linguagem fazem tudo para esconder esse detalhe, e o RUST deu uma importante contribuição que em breve será adotado por várias linguagem.

E mesmo que não exista, ponteiros são constantemente reinventados como uma forma de falta de criatividade e conhecimento da interface padrão de diversas linguagens.
