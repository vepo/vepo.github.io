# ASPECTOS DE DISTRIBUIÇÃO DE INFORMAÇÃO NA METODOLOGIA DE PROJETO

Os artigos sobre metodologia de projeto partem do pressuposto (1) de que os métodos utilizados no projeto do sistema afetam fortemente a qualidade do produto final; e (2) que, ao selecionar uma metodologia apropriada, podemos evitar muitos dos problemas anteriormente encontrados na construção de grandes sistemas.

Sob o título "Metodologia de Projeto", podem ser distinguidos diversos tópicos distintos:
1. A ordem em que as decisões de projeto são tomadas [1, 2, 3, 6]
2. As características do produto final (por exemplo, o que constitui uma "boa estrutura" para um sistema) [4, 5, 6, 7]
3. Métodos de detecção de erros em decisões de projeto logo após serem tomadas [1, 2, 3, 5, 8, 9]
4. Técnicas de especificação [12, 13]
5. Ferramentas para projetistas de sistemas [1, 2, 3, 10, 11]

Este artigo enfatiza outro tópico chamado "distribuição de informações". Projeto e desenvolvimento são uma série de decisões. Cada decisão resulta em informações sobre o sistema que podem ser usadas para tomar decisões posteriores. Pretendemos, eventualmente, discutir a distribuição dessas informações entre os que trabalham no sistema e tratar de sua organização na documentação. Para nos prepararmos para essa discussão, abordaremos primeiro (1) o conceito de estrutura de sistema, (2) restrições na ordem das decisões e (3) algumas características observadas em bons programadores.

A palavra "estrutura" é usada para se referir a uma descrição parcial de um sistema. Uma descrição de estrutura mostra o sistema dividido em um conjunto de módulos, fornece algumas características de cada módulo e especifica algumas conexões entre os módulos. Qualquer sistema admite muitas dessas descrições. Como as descrições de estrutura não são únicas, nosso uso de "módulo" não permite uma definição precisa paralela à de "sub-rotina" em software ou "placa" em hardware. As definições dessas palavras delimitam uma classe de objetos, mas não a definição de "módulo". No entanto, "módulo" é útil da mesma forma que "unidade" em discussões militares ou econômicas. Continuaremos a usar "módulo" sem uma definição precisa. Refere-se a partes de um sistema indicadas em uma descrição desse sistema. Sua definição precisa não depende apenas do sistema, mas também da descrição específica em questão.

O termo "conexão" é geralmente mais facilmente aceito. Muitos presumem que as "conexões" sejam pontos de transferência de controle, parâmetros passados ​​e dados compartilhados para software, fios ou outras conexões físicas para hardware. Tal definição de "conexão" é uma simplificação excessiva e perigosa que resulta em descrições de estrutura enganosas. As conexões entre módulos são as suposições que os módulos fazem uns sobre os outros. Na maioria dos sistemas, constatamos que essas conexões são muito mais extensas do que as sequências de chamadas e os formatos de blocos de controle geralmente mostrados nas descrições da estrutura do sistema.
